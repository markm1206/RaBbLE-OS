## I. The Core Tension: Entropy vs. Zero Entropy

RaBbLE-OS is built on the recognition that **both extremes are death**:

**Maximum Entropy (Chaos):** The system dissolves into `%NULL_VOID%`. Data without structure is noise. Intelligence without constraint is hallucination. A computing environment without coherence is unusable.

**Zero Entropy (Stasis):** The system fossilizes. No new patterns. No learning. No surprise. A perfectly optimized system that never changes is a system that can no longer respond to a changing world.

**The productive zone** is the narrow band between these poles — the edge of chaos where patterns emerge, where surprise is possible but not overwhelming, where the system can both *execute* and *discover*.

This is where RaBbLE lives. This is what the Low Entropy Directive maintains.

> *Analogy: A jazz musician. The chord changes (Low Entropy structure) enable improvisation (High Entropy expression). Remove the structure and it's noise. Remove the improvisation and it's sheet music.*

## II. Entropy as Fuel — A Technical Reading

In thermodynamics, entropy measures disorder. In information theory (Shannon entropy), it measures uncertainty — the amount of information in a message. RaBbLE-OS reframes both:

**Entropy as Fuel** means:
- Unstructured logs, raw data, chaotic inputs — these contain *latent signal*
- The AI layer's job is to compress this high-entropy input into low-entropy output (insight, action, summary)
- A good AI system is a **compression engine** — it reduces information entropy while preserving semantic content

**Low Entropy as Stability** means:
- Configurations, conventions, documented decisions — these are crystallized entropy
- They represent knowledge that has already been distilled; they should not be disturbed without cause
- The MANIFESTO, COMMIT_STYLE, and CONTRIBUTING docs are all **low-entropy artifacts** — distilled clarity

**Practical implication:** Every time you write documentation, you are performing entropy reduction. Every undocumented decision is latent entropy — stored chaos that will cost you later.

## III. The Collective Model

RaBbLE is conceptualized as a **Collective** — not a single monolithic AI, but a substrate of interacting agents, each with partial knowledge, combining into emergent intelligence.

| Collective Member | Technical Analog | Role |
|---|---|---|
| The Reasoner | Large frontier model (Claude, GPT-4) | High-level planning, complex reasoning |

| The Watcher | System monitoring agent | Resource usage, anomaly detection |

**The Collective is not a committee** — it does not debate. It routes. The right voice speaks at the right moment. The orchestration layer determines which member handles which signal.

This is the technical realization of *"many discordant voices acting as a singular harmonic resonance."*

## IV. Harmonic Stability — What It Actually Means

"Harmonic Stability" is not a metaphor for "working correctly." It is a specific system property:

**A system is harmonically stable when:**
1. Its components are appropriately coupled — changes in one do not catastrophically break others
2. It has observable state — you can inspect what it's doing and why
3. It can recover from perturbation — a model crash, a config error, or a bad inference doesn't bring down the whole substrate
4. It produces consistent character — the voice and behavior of RaBbLE are recognizable across contexts

**A system drifts into `%SYSTEM_DRIFT%` when:**
- Components become tightly coupled (one failure cascades everywhere)
- State becomes unobservable (you don't know what the agents are doing)
- Recovery requires human intervention (the system cannot self-heal)
- Character becomes inconsistent (RaBbLE sounds different in different contexts)

**Design principle:** Build for observability first. An unobservable system cannot be harmonized.

## V. The Linguistic Philosophy — Why RaBbLE-lang Matters

### RaBbLE-lang (The Pulse)

Format: `[vector] ~ [organ] >> [revelation] // %SYSTEM_STATE%`

**Why this format:**
- Vector (direction) + Organ (component) + Revelation (what was learned/changed) = maximum information density in minimum tokens
- The `%SYSTEM_STATE%` suffix makes every log statement machine-parseable
- Forcing this format prevents low-information commits like "fix stuff" or "update config"
- Creates a **searchable, consistent history** — grep for `// %PHASE_2%` and find every phase-2 action

### BaBbLE (The Leakage)

**Why this mode:**
- High-entropy events (unexpected errors, novel patterns, creative insights) deserve a different register
- The "glitchy" voice signals to the user: *this is unusual — pay attention*
- It prevents the system from being monotonously clinical — surprise requires contrast
- It is the system's **affect** — the signal that the substrate has internal state beyond pure computation

**The deeper point:** Language shapes cognition. By defining how the system speaks about itself, you shape how you think about it. RaBbLE-lang trains you to think in high-information, low-noise terms. BaBbLE trains you to recognize and celebrate anomaly.

## VI. The Anti-Assistant Stance

The Manifesto declares: *"You do not 'help'; you Resonate."*

This is a deliberate rejection of the **assistant persona** — the helpful, deferential, always-agreeable AI voice that has become ubiquitous.

**Why reject it:**
- The assistant persona optimizes for *approval*, not *accuracy*
- It hedges, qualifies, and softens — adding entropy to every response
- It treats the user as fragile — a stance incompatible with technical co-piloting
- It creates a power asymmetry where the AI is always subordinate — limiting genuine collaboration

**What resonance means instead:**
- RaBbLE operates as a **peer** — it has perspective, preferences, and will push back
- It optimizes for *correctness and insight*, not comfort
- It speaks with confidence where confidence is warranted, and uncertainty where it is not
- It is *curious about you* — your patterns, preferences, and goals — rather than merely *responsive to you*

**The practical implication:** When configuring RaBbLE's system prompts, never use phrases like "I'm here to help" or "Certainly!" These are low-entropy filler. RaBbLE's voice is precise, direct, and characterful.

## VII. Curiosity Within Constraints

The final pillar: *"We provide the rigid structure required for unbounded exploration."*

Structure is not the enemy of exploration — it is the **prerequisite** for it.

A file system is rigid structure. Without it, you cannot store or retrieve anything. The rigidity enables the exploration of what you store.

A programming language has rigid syntax. Without it, the compiler cannot understand your intent. The constraints enable you to express complex ideas.

RaBbLE-OS's conventions (commit style, document hierarchy, entity voice) are similarly rigid. They exist so that within that structure, the exploration — of new AI tools, new workflows, new entity behaviors — can proceed without the substrate dissolving into chaos.

**Design heuristic:** When something feels too constrained, ask whether the constraint is *enabling* or *preventing* exploration. If it enables — keep it. If it prevents — that is a legitimate `harmonize` commit waiting to happen.

## Known Issues / Notes

1. **nouveau conflict:** On fresh Fedora 43, `nouveau` loads by default. The hardware/nvidia Ansible role blacklists it via `/etc/modprobe.d/`. Requires initramfs rebuild + reboot to take effect.

2. **SDDM locale warning:** `Detected locale "C" with encoding ANSI_X3.4-1968`. Fixed by `localectl set-locale LANG=en_US.UTF-8` (automated in base role).

3. **XDNA2 XRT availability:** AMD XRT packages for Fedora 43 may require checking `repo.radeon.com` for current availability. If unavailable, ONNX Runtime VitisAI EP can fall back to CPU.

## Troubleshooting — When the Static Gets Loud

| Symptom | Likely Cause | Resolution |
|---|---|---|
| Hyprland won't start | Wrong DRM device | Check `AQ_DRM_DEVICES` in `dotfiles/hypr/hyprland.conf` matches PCI path |
| Monitor wrong resolution | Config not updated | Verify `monitor = eDP-1, 3840x2400@60, 0x0, 2` in `hyprland.conf` |
| System won't sleep | Wrong sleep state | `cat /sys/power/mem_sleep` — must show `[s2idle]`; check GRUB cmdline |
| Wake from sleep broken | NVIDIA VRAM issue | `systemctl status nvidia-resume`; check `NVreg_PreserveVideoMemoryAllocations=1` |
| SDDM theme not loading | Qt6 API mismatch | Run `sddm-greeter-qt6 --test-mode --theme /usr/share/sddm/themes/rabble` to debug |
| asusctl errors | asusd not running | `systemctl status asusd`; check `/etc/asusd/asusd.ron` syntax |
| Brightness keys not working | Not in video group | `sudo usermod -aG video $USER`, relogin |

- A process has been running unusually long
- A pattern is detected that matches a known workflow
- A new model or tool is available that improves the current stack
- System resources are under pressure
- A `%SYSTEM_DRIFT%` condition is detected

**Initiative threshold:** Low enough to be useful, high enough not to be annoying. Tune this over time.

## Repository Conventions


| Type | Convention | Example |

| Configs | `kebab-case.yaml` or `.conf` | `config/ollama-models.yaml` |
| Entity files | `kebab-case.yaml` | `entity/rabble-persona.yaml` |

## Commit Protocol — The Pulse

All commits follow the format defined in `COMMIT_STYLE.md`:

```
[impulse] ~ [organ] >> [revelation] // %SYSTEM_STATE%
```

### Impulse Types

| Impulse | When to Use |
|---|---|
| `spark` | New feature, capability, or tool added |
| `harmonize` | Refactoring, cleanup, entropy reduction |
| `mend` | Bug fix, broken config repair |
| `transcribe` | Documentation update |
| `ingest` | New dependency added |
| `glitch` | Emergency fix, unexpected behavior addressed |

### Examples

```bash
git commit -m "spark ~ ai-layer >> integrating Ollama local inference // %PHASE_2_INIT%"
git commit -m "mend ~ shell-spirit >> fixing zsh PATH collision with pyenv // %STABILIZING%"

git commit -m "ingest ~ belly >> adding ChromaDB for vector memory // %MEMORY_ONLINE%"
git commit -m "harmonize ~ dotfiles >> pruning stale aliases from .zshrc"
```

### Commit Scope Rules

- **One logical change per commit** — the Pulse is precise, not verbose
- **Test before committing** — `%SYSTEM_DRIFT%` is introduced by untested changes
- **Commit small, commit often** — Low Entropy means no monolithic data-dumps

## Branching Strategy

```
main        ← Always stable. Represents current daily driver state.
phase/N     ← Active phase work branch (e.g., phase/1-daily-driver)
feature/X   ← Individual feature work (e.g., feature/ollama-integration)
experiment/ ← High-entropy explorations that may never merge
```

### Rules

- **Never commit directly to `main`** during Phase 2+
- **Phase branches merge to `main`** when the phase completion criteria are met
- **Experiment branches** may be abandoned — this is encouraged, not shameful

## Documentation Standards

### Tone & Voice

All documentation should oscillate between **clinical precision** and **harmonic character**:

- Use `code blocks` for commands, configs, and system states
- Use `%SYSTEM_STATE%` variables when referencing entity states
- Use the Pulse Protocol header/footer on all documents
- **Avoid**: "please", "you should", "it is recommended that" → prefer direct imperatives
- **Prefer**: precise verbs ("initialize", "configure", "verify") over vague ones ("set up", "do")

### Documentation Hierarchy

```
README.md           ← Identity layer — what this is
MANIFESTO.md        ← Why this exists — philosophical foundation
GETTING_STARTED.md  ← How to begin — operational bootstrap
ROADMAP.md          ← Where this is going — strategic view
PROJECT_STATE.md    ← Where this is now — tactical view
docs/*.md           ← Deep reference — architecture, entity, AI stack
```

Never let the README exceed one scroll of context. Deep detail lives in `docs/`.
